## 执行摘要

本指南定义了技术文档的统一写作风格和中英文双语发布标准。通过遵循这些规范，我们确保所有技术内容保持专业性、一致性和可读性，同时服务于全球安全社区。

## 1. 核心写作原则

### 1.1 目标受众定位
- **主要受众**：网络安全从业者、技术决策者、安全研究人员
- **技术水平**：中高级，具备基础安全知识
- **阅读目的**：获取技术洞察、解决方案指导、威胁情报

### 1.2 写作语调
- **专业但不晦涩**：使用准确的技术术语，但避免过度学术化
- **客观且权威**：基于事实和数据，避免主观臆断
- **实用导向**：强调实际应用价值和可操作性

### 1.3 内容质量标准
- **原创性**：提供独特见解和原创研究
- **准确性**：技术细节经过验证，数据来源可靠
- **时效性**：反映最新技术发展和威胁态势
- **完整性**：覆盖主题的关键方面，不遗漏重要信息

## 3. 中英文双语规范

### 3.1 翻译原则
- **内容完整对应**：中英文版本信息量保持一致
- **术语标准化**：建立术语对照表，确保一致性
- **文化适应性**：考虑不同地区读者的理解习惯

### 3.2 文件命名规范
- 中文版：`[主题名称]_中文版.md` 或 `[主题名称]_CN.md`
- 英文版：`[主题名称]_EN.md` 或 `[Topic_Name]_EN.md`

### 3.3 专业术语处理
- **保留英文缩写**：APT、SIEM、SOC等广泛认可的缩写
- **中文解释并用**：首次出现时提供中文解释
- **统一术语表**：维护项目级术语对照表

### 3.4 代码和命令
- 代码块保持原文，不翻译
- 注释使用对应语言版本
- 命令说明使用文档主语言

## 4. 格式规范

### 4.1 标题层级
- 一级标题：文章标题
- 二级标题：主要章节
- 三级标题：子章节
- 避免超过四级标题

### 4.2 段落结构
- 每段专注一个主题
- 段落长度控制在3-5句
- 使用过渡句连接段落

### 4.3 列表使用
- 无序列表：并列关系的要点
- 有序列表：步骤或优先级关系
- 嵌套不超过两层

### 4.4 强调方式
- **粗体**：关键概念、重要结论
- *斜体*：首次引入的术语、引用
- `代码格式`：命令、函数名、变量

### 4.5 数据展示
- 使用表格展示对比数据
- 百分比保留小数点后一位
- 大数字使用千位分隔符

## 5. LinkedIn专栏发布指南

### 5.1 标题优化
- 突出技术关键词
- 控制在60个字符以内
- 包含价值主张

### 5.2 内容调整
- 开篇使用吸引人的引言
- 适当缩短技术深度
- 增加视觉元素说明

### 5.3 互动元素
- 文末提出讨论问题
- 邀请读者分享经验
- 提供延伸阅读链接

### 5.4 内容格式优先级（2025年数据）

基于2025年LinkedIn平台表现数据，不同格式的平均互动率如下：

| 内容格式 | 平均互动率 | 推荐使用场景 |
|---------|-----------|-------------|
| **多图片帖子** | 6.60% | 技术架构图解、流程对比、案例展示 |
| **原生文档** | 5.85% | 深度技术报告、白皮书摘要 |

**格式选择建议**：
- **优先原生格式**：直接上传图片、文档、视频，而非外部链接
- **避免纯文本**：至少配合1-3张相关图片或图表
- **视频趋势**：2025年视频使用量增长69%，覆盖率提升23%，互动率提升20%

### 5.5 算法优化策略（2025年更新）

LinkedIn 2025年算法三大核心因素：

**1. 主题相关性（Topical Relevance）**
- 强调专业性和原创见解
- 包含行业趋势分析和可操作建议
- 使用相关话题标签（#Cybersecurity #AIInSecurity #ThreatIntelligence）

**2. 早期互动速度（Early Engagement Velocity）**
- **黄金窗口期**：发布后前60-120分钟
- 评论（Comments）权重 > 点赞（Likes）> 分享（Shares）
- 建议：发布后主动回复前5-10条评论，激活算法推荐

**3. 观众质量信号（Viewer Quality Signals）**
- 内容被行业专家互动
- 同领域从业者的持续关注
- 真实对话 > 表面互动

**内容策略调整**：
- ✅ **真实性优先**：分享经验和实际案例，而非泛泛的统计数据
- ✅ **人性化呈现**：讲故事、展示过程，避免纯推广性质的内容
- ❌ **减少外部链接**：外链帖子互动率显著低于原生内容

### 5.6 发布频率与时机

**推荐发布频率**：
- **活跃账号**：每周3-5次（保持持续可见性）
- **新账号建立期**：每周5-7次（快速建立影响力）
- **成熟账号**：每周2-3次（质量优于数量）

**内容类型分布建议**：
- 40%：行业洞察和趋势分析
- 30%：技术深度文章（长文）
- 20%：案例研究和实战经验
- 10%：社区互动（投票、讨论话题）

**时机考量**：
- 优先工作日发布（周二至周四表现最佳）
- 避免周末发布技术深度内容
- 考虑目标受众的时区（中文内容：北京时间9-11AM、3-5PM）

### 7.5 AI辅助写作协作流程（2025年指南）

基于2025年AI辅助技术写作最佳实践，建立**人机协作（Human-in-the-Loop）**工作流程。

#### 7.5.1 AI工具应用场景

**推荐使用AI辅助的场景**：
- **初稿生成**：根据大纲快速生成文章框架
- **语言优化**：改进句式结构和表达流畅度
- **多语言翻译**：生成初版英文翻译（需人工审校）
- **代码示例**：生成基础代码框架（必须测试验证）
- **摘要生成**：从长文中提取关键信息

**禁止完全依赖AI的场景**：
- ❌ 技术准确性判断（必须人工验证）
- ❌ 数据来源核实（AI可能虚构来源）
- ❌ 最终发布决策（需人工质量把关）
- ❌ 敏感信息处理（避免泄露机密）

#### 7.5.2 Human-in-the-Loop验证机制

**四层验证流程**：

**第1层：AI生成**
- 使用AI工具生成初稿或建议
- 明确提示词要求（如"生成一篇关于零信任架构的技术分析，包含执行摘要、技术原理、实施步骤"）

**第2层：技术准确性验证**
- ✅ **多源交叉验证**：AI提及的技术概念必须与≥3个权威来源核对
- ✅ **代码测试**：所有代码示例必须在真实环境中运行通过
- ✅ **数据核实**：检查统计数据的真实性和时效性

**第3层：风格一致性审校**
- 参照Microsoft Style Guide或项目风格指南
- 确保术语使用与项目术语表一致
- 检查是否符合Plain Language原则

**第4层：最终人工审核**
- 完整通读，确保逻辑连贯
- 使用ARTICLE_ACCURACY_CHECKLIST.md逐项检查
- 考虑目标受众的理解程度

#### 7.5.3 AI事实核查工具链

**推荐工具组合**：

| 工具类型 | 推荐工具 | 用途 |
|---------|---------|------|
| **事实核查** | Perplexity AI | 验证AI生成的声明和数据 |
| **语法检查** | Grammarly Pro | 英文语法和风格优化 |

#### 7.5.4 AI生成内容标注原则

**透明度要求**：
- 在使用AI辅助生成的文章中，无需逐句标注
- 关键技术分析和原创观点必须来自人类专家
- 代码示例若由AI生成，需标注"代码示例经AI辅助生成并经过测试验证"

**质量基准**：
- AI辅助内容必须达到与人工撰写相同的质量标准
- 发布前必须通过完整的准确性验证流程
- 对AI生成的内容负全责，不得以"AI生成"为错误开脱

#### 7.5.5 实际应用示例

**场景：撰写威胁情报分析文章**

```

- 运行准确性检查清单
```

### 7.6 人类化写作规范（2026年强制要求）

> **重要更新（v3.0 - 2026-01-04）**：人类化写作已升级为**强制默认要求**，所有输出内容必须通过人类化验证。

#### 7.6.1 理论基础：Perplexity与Burstiness

**Perplexity（困惑度）**：
- 定义：衡量文本可预测性的指标
- AI特征：倾向选择高频、"安全"的词汇，导致低困惑度
- 人类特征：使用个性化、低频词汇，困惑度更高

**Burstiness（突发性）**：
- 定义：衡量句子长度和复杂度变化的指标
- AI特征：句子长度趋于均匀（低突发性）
- 人类特征：长短句自然交替（高突发性）

#### 7.6.2 强制指标

| 指标 | 最低要求 | 理想值 | 检测方法 |
|------|----------|--------|----------|
| Burstiness指数 | ≥0.7 | ≥0.9 | 句子长度标准差/均值 |
| AI话术数量 | ≤3处 | 0处 | 关键词匹配 |
| 高频词重复 | 单词≤5次/千字 | ≤3次/千字 | 词频统计 |
| 短段落占比 | ≤55% | ≤40% | 段落长度分析 |

#### 7.6.3 AI话术黑名单（绝对禁止）

```
🚫 以下表达禁止使用：
- "让我们深入探讨..."
- "值得注意的是..."
- "综上所述..." / "总的来说..."
- "不可否认..." / "显而易见..."
- "毫无疑问..."
- "首先...其次...最后..."（三段式）
- "在当今...的时代"
- "我们可以得出结论..."
```

**替代方案**：
| 避免 | 替换为 |
|------|--------|
| 让我们深入探讨 | 来聊聊 / 说说 / 看看 |
| 值得注意的是 | 有意思的是 / 说到这 / 顺便提一句 |
| 总的来说 | 说到底 / 归根结底 / 总之 |
| 综上所述 | 所以呢 / 归纳一下 / 简单说 |
| 不可否认 | 确实 / 没错 / 这点没得说 |

#### 7.6.4 句式变化技巧（提升Burstiness）

```markdown
❌ AI风格（句子长度均匀）：
Raycast是一款高效的生产力工具。它提供了丰富的扩展生态系统。
用户可以通过快捷键快速访问各种功能。这显著提升了日常工作效率。

✅ 人类风格（长短交替）：
Raycast改变了我的工作方式。不夸张地说，这款启动器让我每天
```

## 8. Plain Language实施指南（2025年最佳实践）

### 8.1 Plain Language核心原则

基于SANS、CISA等权威机构的技术写作指南，Plain Language（简明语言）是提升网络安全文档可读性和可操作性的关键。

**核心目标**：
- 让非专家受众也能理解技术概念
- 减少因误解导致的安全配置错误
- 提高安全策略的执行效率

### 8.2 语言简化策略

**避免过度技术化**：
| ❌ 避免使用 | ✅ 推荐使用 | 适用场景 |
|------------|-----------|---------|
| "实施纵深防御机制" | "部署多层安全防护" | 面向管理层 |
| "建立基于角色的访问控制矩阵" | "根据职责分配系统权限" | 操作指南 |
| "部署SIEM平台进行日志聚合与关联分析" | "收集并分析安全日志，识别异常活动" | 技术说明 |
| "配置DMARC、SPF、DKIM三重邮件认证" | "设置邮件验证，防止伪造发件人" | 执行步骤 |

**句式简化原则**：
- **一句一意**：每个句子只表达一个主要观点
- **主动语态**：使用"攻击者利用漏洞"而非"漏洞被攻击者利用"
- **简短句子**：控制在15-20个词以内
- **去除冗余**：删除"基本上"、"事实上"等填充词

### 8.3 结构化呈现

**流程图和图表应用**：
- **决策流程**：使用流程图展示"发现安全事件后的响应步骤"
- **对比分析**：使用表格对比"传统防火墙vs下一代防火墙"
- **架构说明**：使用架构图展示"零信任网络架构的组件关系"

**分步骤说明**：
```markdown
## 配置双因素认证（2FA）

### 步骤1：选择认证方式
- 推荐：基于时间的一次性密码（TOTP）应用（如Google Authenticator）
- 备选：短信验证码（安全性较低，但易用性高）

### 步骤2：启用2FA
1. 登录管理后台
2. 导航至"安全设置" > "双因素认证"
3. 扫描二维码或输入密钥
4. 输入6位验证码确认

### 步骤3：保存恢复代码
⚠️ 重要：将恢复代码保存到安全位置，避免失去设备时无法登录
```

### 8.4 文化适应性

**避免习语和地区特定表达**：
- ❌ "这个漏洞是个定时炸弹"（中文习语）
- ✅ "这个漏洞如果不修复，可能在未来导致严重安全事件"

- ❌ "Let's touch base on the security audit"（英文习语）
- ✅ "Let's discuss the security audit progress"

**通用案例描述**：
- 使用"某金融机构"而非具体公司名
- 使用"亚太地区某制造企业"等地域化但非特定的描述

### 8.5 Plain Language检查清单

**发布前自查**：
- [ ] 每个段落是否聚焦单一主题？
- [ ] 是否将复杂流程分解为3-5个步骤？

- [ ] 是否避免了习语、俚语和文化特定表达？

- [ ] 是否使用了主动语态？
- [ ] 关键操作是否使用了编号列表？

**工具推荐**：
- **可读性测试**：Hemingway Editor（英文）
- **术语一致性**：建立项目术语表，使用搜索功能核查
- **用户测试**：请非技术团队成员试读，收集反馈

### 8.6 技术深度与可读性平衡

**分层呈现策略**：

**层次1 - 执行摘要**（适合管理层）：
- 200-300字概述
- 强调业务影响和关键决策点
- 使用Plain Language

**层次2 - 技术概述**（适合技术决策者）：
- 1000-1500字详细说明
- 包含架构图和关键技术选型
- 平衡专业性与可读性

**层次3 - 实施细节**（适合执行工程师）：
- 完整配置步骤和代码示例
- 允许使用专业术语
- 提供详细注释

**示例**：
```
[执行摘要] 本文提出基于零信任架构的远程访问方案，可降低数据泄露风险。

[技术概述] 零信任架构（Zero Trust Architecture, ZTA）采用"永不信任，始终验证"的原则...

[实施细节] 步骤1：配置身份验证服务器（Identity Provider, IdP）
```

## 9. 常见问题

### Q: 如何平衡技术深度和可读性？
A: 使用分层结构，执行摘要提供概览，正文逐步深入，技术细节放在附录或专门章节。

### Q: 中英文版本差异如何处理？
A: 保持核心内容一致，可以根据目标受众微调案例和解释方式，但技术信息必须完全对应。

### Q: 如何确保内容的时效性？
A: 建立定期审查机制，及时更新过时信息，在文档中标注最后更新时间。
